Next up in Order: Seasons

I’d like to develop the concept of a season (see Atlas Journaling — A Unified Approach for Seeing the Big Picture and Acting with Focus)

The idea is that in a calendar view, there will be new option after year called ‘Season’.

A season has a user defined start and end point and a visualization:

Imagine the Areas grid take up the whole screen. In my case it will be a 3 row by 2 column grid. Each cell in the grid representing an area will have a list of notable folders for the season. The arrows will move between seasons much like the arrows move between days weeks, months, and years.

How does a user define a season and how is it represented on disk?

First, there is Areas.md that has a special role YAML frontmatter to distinguish it as special.

First, a season is a first class concept in Order. If role is seasons (for the standard case it will be Seasons.md in the root of the vault. This file is just a markdown file with a list of ISO 8601 date ranges in a format like:

- 2026-01-27 - 2026-05-24 · Optional name of season
- 2026-06-01 -  · Current season (no determined end yet)

The idea is that if there exists a notable update (all day event) for a notable folder in this season's date range, then the corresponding parent Area the notable folder is a part of is populated with a link to that folder. The notable folders are in sorted within an area box in order of date of most recent all day event associated with the notable folder.
