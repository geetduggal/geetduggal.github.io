Feature idea, logged and deferred.

Goal

For a note or Notable Folder with list: cards or list: lines, stop collapsing the whole body into one list. Instead, render the body in document order: keep the prose blocks as they are, and turn each consecutive run of bullet items into its own separate visual list (cards or lines). Text between two runs of bullets stays put, and each bullet group becomes an independent list.

Current behavior

splitBodyAndBullets pulls every bullet out of the body and renders them as a single list at the bottom, dropping the surrounding prose from the list view. So separate groups of bullets get merged into one list and the interleaved text is lost in that view.

Decisions made during design

  1. Prose between lists should be fully editable inline (not read only).
  2. Cross group drag is allowed (an item can move from one group into another, which rewrites its position across the intervening text on save).

Key constraint

The editor uses @milkdown/crepe plus @milkdown/kit only. There is no @milkdown/react, and Crepe is a batteries included wrapper that does not cleanly expose node view configuration. This is what makes the two ambitious decisions above expensive.

Architecture options considered

A. One Milkdown editor, lists as ProseMirror node views. The body stays a single editable document; a custom node view renders each bullet run as the cards or lines UI in place. Cleanest fit for inline editing and cross group moves, since it is all one document. Blocked by the missing React node view support; would need @milkdown/react wired into Crepe, or a vanilla DOM rewrite of the list UI. High risk, multi day.

B. Multiple Milkdown editors interleaved with list components. Split the body into ordered segments (prose run, bullet run, prose run, and so on); one editor per prose run, one list component per bullet run. No new dependencies, reuses existing ListCards and ListLines. But inline editing gets awkward at boundaries (typing a bullet into a prose editor), and cross group drag does not exist today (useTileDrag is single grid), so it would need a new cross list drag mechanism. Medium to high effort.

Status

Deferred. Pick this up by choosing an architecture, then writing a proper implementation plan. If we commit to option A, first add @milkdown/react and spike whether Crepe will host a React node view before building.
